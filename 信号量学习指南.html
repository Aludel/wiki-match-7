信号量 Semaphore：从零开始的交互式学习

这份指南把 OSTEP 第 31 章拆成「概念 → 类比 → 流程图 → 互动 → 自测」五步走。即使你没学过锁和条件变量，也能跟着走完。带 ★ 的部分可以动手点。

0开始之前：三个必须先懂的词

在碰信号量之前，先把三个底层概念用大白话说清楚。它们是后面一切的地基。

🧵 线程（Thread）

把一个程序想象成一家餐厅。线程就是餐厅里干活的服务员。一家餐厅可以同时雇好几个服务员（多线程），他们同时在大厅里跑来跑去干活。多个服务员一起干活，效率高，但也会撞车——这就是「并发」带来的麻烦。

🚪 临界区（Critical Section）

餐厅里有一间只能容一个人进去的小储藏室（比如同一时刻只能一个人改账本）。如果两个服务员同时冲进去改账本，账目就乱了。这段「同一时刻最多只能一个线程进入」的代码，就叫临界区。我们需要一把「锁」来管这扇门。

⏳ 等待 / 唤醒（Sleep / Wakeup）

如果储藏室有人，第二个服务员不该站在门口空转浪费体力（这叫忙等待 busy-waiting，很费 CPU）。更聪明的做法是：先去休息室睡一觉，等里面的人出来时拍他一下叫醒他。信号量内部正是用「睡眠 / 唤醒」来省 CPU 的。

好了，地基有了。信号量要解决的核心问题是：多个线程同时干活时，如何协调谁先谁后、谁能进、谁要等。 历史上，Edsger Dijkstra 发明了信号量，把它作为同步的唯一原语——意思是：有了它，锁和条件变量都能用它做出来。

1信号量的定义

信号量就是一个带整数值的对象，外加两个操作函数。在 POSIX 标准里这两个函数叫 sem_wait() 和 sem_post()。使用前必须先初始化它的值：

#include <semaphore.h>
sem_t s;
sem_init(&s, 0, 1);   // 把信号量 s 的值初始化为 1
  • 第 1 个参数 &s：要初始化哪个信号量。
  • 第 2 个参数 0：表示这个信号量在同一个进程的多个线程之间共享（本章所有例子都填 0）。
  • 第 3 个参数（这里是 1）：初始值。这个值至关重要——它直接决定信号量的行为。后面每个问题我们都要反复问「初值该填几？」
🔑 一句话记住

信号量初值 X 大致可以理解为：「现在还有几个名额 / 资源可以用」。初值 1 = 一个名额（典型的锁）；初值 0 = 没名额，谁来都得先等别人给（典型的条件变量 / 等待）。

2wait / post 到底做了什么

书里给的「定义」其实就是两段伪代码，先把它背下来，再理解：

int sem_wait(sem_t *s) {
    将 s 的值 减 1;
    如果 s 的值为负，则当前线程睡眠等待;
}

int sem_post(sem_t *s) {
    将 s 的值 加 1;
    如果有一个或多个线程在等待，唤醒其中一个;
}

🔑 最重要的恒等式（务必理解）

当信号量的值变成负数时，这个负数的绝对值恰好等于正在排队等待的线程个数。

例如值为 -3，说明此刻有 3 个线程睡着了在等。这个关系记住了，几乎所有信号量题目你都能「心算」出来。下面的模拟器会让你亲眼看到这一点。

❓ 自测：sem_post 一定会唤醒某个线程吗？
点击看答案 不一定。post 只是「值 +1」，仅当有线程正在等待时才唤醒一个。如果没人等，它就只是把值加上去（让出一个名额给将来的人）。这也解释了为什么 post 从不阻塞、从不等待条件。

3二值信号量（用作锁）

信号量的第一种用法，就是当锁。把临界区用一对 wait/post 包起来：

sem_t m;
sem_init(&m, 0, 1);    // 初值 X 该填几？ → 答案是 1

sem_wait(&m);          // 上锁（进门）
// 临界区 critical section
sem_post(&m);          // 解锁（出门）
🔑 为什么初值是 1？

因为锁只有两个状态：被持有 / 空闲——所以这种用法叫二值信号量（binary semaphore）。初值 1 表示「门开着，一个名额」。第一个线程 wait 后值变 0（占住），它在临界区时第二个线程来 wait 会变 -1 而睡着，等第一个线程 post 把值加回去并唤醒它。

❓ 自测：如果有 3 个线程同时排队等这把锁，信号量的值会是多少？
点击看答案 值会是 -3。一个线程持锁（把值从 1 减到 0），随后 3 个线程各做一次 wait，把值依次减到 -1、-2、-3。负数的绝对值 = 排队等待的线程数，正好 3 个。可以在上面的模拟器里验证。

4信号量用作条件变量（一个等一个）

第二种用法：让一个线程停下来等某件事发生，另一个线程把事做完后发信号叫醒它。经典例子：父线程创建子线程，并等子线程跑完再继续。

sem_t s;

void *child(void *arg) {
    printf("child\n");
    sem_post(&s);     // 子线程：我做完啦，发信号
    return NULL;
}
int main() {
    sem_init(&s, 0, 0);   // 初值 X 该填几？ → 答案是 0
    printf("parent: begin\n");
    Pthread_create(&c, NULL, child, NULL);
    sem_wait(&s);     // 父线程：在这里等子线程
    printf("parent: end\n");
}

期望输出顺序固定为：parent: begin → child → parent: end。

🔑 为什么初值是 0？

因为我们希望「父亲必须等」。初值 0 意味着一开始没有名额，谁先 wait 谁就得睡。这样无论哪种执行顺序，结果都对。下面看两种情况，初值 0 都能兜住。

情况一：父先跑（子线程还没动）

父：sem_wait() 值 0→−1
↓
值 < 0 → 父睡眠 💤
↓
子线程开始运行打印 child
↓
子：sem_post() 值 −1→0唤醒父亲 ⏰
↓
父醒来，wait 返回打印 parent: end

情况二：子先跑完（父还没 wait）

子线程先运行打印 child
↓
子：sem_post() 值 0→1没人在等，不唤醒
↓
父亲后来才运行
↓
父：sem_wait() 值 1→0
↓
值 ≥ 0 → 不用睡，直接返回打印 parent: end

图：初值 0 让两种调度顺序都得到正确结果——这就是设计的精妙之处

💡 类比

把 post 想成「往储蓄罐里投一枚硬币」，wait 想成「从罐里取一枚硬币，没有就在旁边等」。初值 0 = 罐子空的。父亲想取硬币，要么子线程先投了（情况二，直接取走），要么父亲先到只能干等到子线程来投（情况一）。

5生产者 / 消费者（有界缓冲区）

这是最经典的并发问题，也是面试常客。场景：有一个容量有限的缓冲区（数组）。生产者往里放数据，消费者从里取数据。规则很自然：

  • 缓冲区满了，生产者必须等（等消费者腾出空位）。
  • 缓冲区空了，消费者必须等（等生产者放点东西）。

第一步：用两个信号量 empty 和 full

empty 记「还有几个空位」，初值 = MAX；full 记「有几个数据可取」，初值 = 0。

sem_t empty, full;

void *producer(void *arg) {
  for (i=0; i<loops; i++) {
    sem_wait(&empty);  // 占一个空位 (没空位就等)
    put(i);           // 放数据
    sem_post(&full);   // 多了一个可取的数据
  }
}
void *consumer(void *arg) {
  while (...) {
    sem_wait(&full);   // 取一个数据 (没数据就等)
    tmp = get();        // 拿数据
    sem_post(&empty);  // 多了一个空位
  }
}
// main:
sem_init(&empty, 0, MAX);  // 开始全是空位
sem_init(&full,  0, 0);    // 开始没有数据

当 MAX=1 时这段代码完全正确。但当 MAX>1、且有多个生产者/消费者时，问题来了——竞态条件。

⚠️ 问题一：竞态条件（数据被覆盖）

两个生产者 Pa、Pb 几乎同时调用 put()。Pa 把数据写到 buffer[0]，但还没来得及把下标 fill 加 1 就被切走；Pb 接着运行，也往 buffer[0] 写——Pa 的数据被覆盖丢失了！ 根因：往缓冲区写数据和更新下标，是一段临界区，却没有保护。

第二步（错误尝试）：加一把锁，但锁太大

很自然的想法：用一个 mutex（初值 1）把整段包起来。但如果把 mutex 放在最外层（包住 empty/full 的 wait），会死锁：

消费者：sem_wait(mutex)✅ 拿到锁
↓
消费者：sem_wait(full)缓冲区空，没数据
↓
消费者睡眠 💤❗但手里还攥着 mutex
生产者：sem_wait(mutex)❌ 锁被消费者占着，拿不到
↓
生产者也睡眠 💤
🔄 循环等待：消费者抱着锁等生产者发 full　↔　生产者要锁才能造数据 → 互相卡死 = 死锁

图：循环等待 —— 消费者抱着锁睡等数据，生产者要锁才能造数据。互相卡死 = 死锁

🔑 死锁的本质

消费者持有 mutex 却去等 full；生产者能发 full 却卡在等 mutex。两者互相等对方手里的东西，谁也动不了。这叫循环等待，是死锁的典型成因。

第三步（正确方案）：缩小锁的范围

解决办法只有一句话：把 mutex 的 wait/post 挪进去，只包住真正的临界区（put/get），让 empty/full 的等待留在锁外面。

void *producer(void *arg) {
  for (i=0; i<loops; i++) {
    sem_wait(&empty);        // ← 等空位 (在锁外面)
    sem_wait(&mutex);        // ← 上锁 (挪到这里)
    put(i);
    sem_post(&mutex);        // ← 解锁
    sem_post(&full);         // ← 通知有数据 (在锁外面)
  }
}
void *consumer(void *arg) {
  for (i=0; i<loops; i++) {
    sem_wait(&full);         // ← 等数据 (在锁外面)
    sem_wait(&mutex);        // ← 上锁
    tmp = get();
    sem_post(&mutex);        // ← 解锁
    sem_post(&empty);        // ← 通知有空位 (在锁外面)
  }
}
// main: empty=MAX, full=0, mutex=1
✅ 记忆口诀

「先等条件，再上锁；先解锁，再发信号。」 empty/full（条件）在外圈，mutex（互斥）在内圈，紧贴临界区。这样线程睡觉时手里不攥着锁，死锁就解开了。这是多线程程序的常用模式，一定要记牢。

wait(empty)等到空位
→
wait(mutex)上锁
→
put()临界区
→
post(mutex)解锁
→
post(full)通知有数据

图：正确结构 —— 条件信号量在外，互斥锁紧贴临界区在内

6读者—写者锁

动机：有些数据结构（比如链表），查找（读）可以很多人同时做，只有插入/修改（写）才需要独占。普通锁太死板（读也只能一个一个来），于是有了更灵活的读者—写者锁：允许「多个读者并发」或「一个写者独占」。

typedef struct {
  sem_t lock;        // 保护 readers 计数的小锁, 初值1
  sem_t writelock;   // 真正的"独占权", 初值1
  int readers;       // 当前有几个读者在读
} rwlock_t;

void acquire_readlock(rwlock_t *rw) {
  sem_wait(&rw->lock);
  rw->readers++;
  if (rw->readers == 1)        // 我是第一个读者
    sem_wait(&rw->writelock);   // → 替所有读者抢下写锁, 挡住写者
  sem_post(&rw->lock);
}
void release_readlock(rwlock_t *rw) {
  sem_wait(&rw->lock);
  rw->readers--;
  if (rw->readers == 0)        // 我是最后一个读者
    sem_post(&rw->writelock);   // → 把写锁还回去, 放写者进来
  sem_post(&rw->lock);
}
void acquire_writelock(rwlock_t *rw) { sem_wait(&rw->writelock); }
void release_writelock(rwlock_t *rw) { sem_post(&rw->writelock); }
🔑 核心机巧：第一个读者「代表大家」抢写锁

只有第一个进来的读者去 sem_wait(writelock)，把写者挡在门外；之后的读者直接进（因为写锁已经被读者群体握住了）。直到最后一个读者离开，才 sem_post(writelock) 把权限还回去。这样就实现了「读者们要么一起在里面、要么写者一个人在里面」。

第 1 个读者到达
↓
readers: 0→1，抢下 writelock 🔒
↓
第 2、3… 个读者readers++ 直接进入读
↓
读者陆续离开 readers−−
↓
readers == 0 ?
是（最后一个走）
↓
释放 writelock 🔓等待的写者可进入
否（还有人在读）
↓
写者继续被挡在外面
⚠️ 公平性缺陷：写者可能饿死

只要源源不断有读者进来，writelock 就一直被读者群体握着，写者永远轮不上，这叫「写者饥饿（starvation）」。更复杂的实现会在「有写者等待时，禁止新读者进入」来缓解，但代价是更复杂。

💡 Hill 定律：大而笨更好

OSTEP 在这里提醒：读者—写者锁听起来很酷，但很复杂，复杂往往意味着慢。很多场景下，简单的普通锁反而更快更省心。「先尝试简单的笨办法」是个好习惯。

7哲学家就餐问题

5 位哲学家围着圆桌，每两人之间放一把餐叉（共 5 把）。哲学家要么思考，要么吃饭；必须同时拿到左右两把餐叉才能吃。如何让大家都能吃上、不死锁、不饿死？

每位哲学家的循环：think() → getforks() → eat() → putforks()。用 5 个信号量代表 5 把叉子，每个初值 1（叉子可用）。辅助函数：left(p)=p，right(p)=(p+1)%5。

有问题的方案（会死锁）

void getforks() {
  sem_wait(forks[left(p)]);   // 先拿左叉
  sem_wait(forks[right(p)]);  // 再拿右叉
}
⚠️ 死锁场景

假如5 个人同时各自拿起了左手边的叉子，那么每个人都举着一把叉、都在等右手边那把——而右手边那把正被邻居举着。所有人都卡住，谁也吃不成。

P0 持叉0等叉1
→
P1 持叉1等叉2
→
P2 持叉2等叉3
→
P3 持叉3等叉4
→
P4 持叉4等叉0
↩ P4 等的「叉0」正被 P0 占着 → 首尾相连成环 → 所有人都卡住 = 死锁

图：每个人都等下一个人手里的叉 → 首尾相连的循环等待 → 死锁

解决方案：破除依赖（打破循环）

Dijkstra 的办法很巧：让编号最大的那位哲学家（P4）反过来，先拿右叉再拿左叉。只要有一个人的拿叉顺序不同，那条「首尾相连的环」就断了。

void getforks() {
  if (p == 4) {                  // 最后一个哲学家
    sem_wait(forks[right(p)]);  // 先右
    sem_wait(forks[left(p)]);   // 后左
  } else {
    sem_wait(forks[left(p)]);   // 其他人: 先左后右
    sem_wait(forks[right(p)]);
  }
}
✅ 为什么有效

因为 P4 会先去抢「叉0」（它的右叉），而 P0 也想要叉0（它的左叉）。两人会争夺同一把叉，于是不可能出现「人手一把、整桌成环」的局面——环被打破，就不会死锁。这其实是「破除循环等待」这一通用反死锁策略的具体应用。

类似的「著名」问题还有吸烟者问题、理发师问题，本质都是用信号量协调线程，名字吸引人而已。

8反过来：用锁+条件变量实现信号量

前面都是「用信号量做别的东西」。最后反向思考：如果手里只有锁和条件变量，能不能造一个信号量出来？能，叫它 Zemaphore。

typedef struct {
  int value;
  pthread_cond_t cond;
  pthread_mutex_t lock;
} Zem_t;

void Zem_init(Zem_t *s, int value) {
  s->value = value;
  Cond_init(&s->cond);
  Mutex_init(&s->lock);
}
void Zem_wait(Zem_t *s) {
  Mutex_lock(&s->lock);
  while (s->value <= 0)            // 注意是 while, 不是 if
    Cond_wait(&s->cond, &s->lock);  // 值≤0 就睡, 醒来重新检查
  s->value--;
  Mutex_unlock(&s->lock);
}
void Zem_post(Zem_t *s) {
  Mutex_lock(&s->lock);
  s->value++;
  Cond_signal(&s->cond);          // 叫醒一个等待者
  Mutex_unlock(&s->lock);
}

🔑 两个易错点

① 为什么用 while 而不是 if？ 线程被唤醒后不能假设条件一定满足（可能被别的线程抢先把值又减没了），所以要重新检查。这是用条件变量的铁律。

② 与 Dijkstra 原版的小区别： 这个实现里 value 永远不会小于 0（不像原始信号量用负值表示等待线程数）。这种行为更好实现，也和现代 Linux 的实现一致。

⚠️ 反向更难

有趣的是：用信号量实现锁和条件变量很容易，但用信号量去实现条件变量却出奇地难，历史上很多老手在 Windows 上栽过跟头。Lampson 的告诫：「不要泛化，泛化通常都是错的。」把信号量当成锁和条件变量的「万能泛化」，未必真的那么通用。

✅ 五条核心原则
  • 恒等式：信号量为负时，|值| = 正在等待的线程数。
  • 初值决定一切：1 = 锁；0 = 等待；MAX = 资源池容量。
  • wait 可能睡，post 从不睡：post 只在有人等时才唤醒。
  • 反死锁：条件信号量在外圈，互斥锁紧贴临界区在内圈；睡觉别攥着锁。
  • 破除循环等待：哲学家问题靠改一个人的拿叉顺序打破环。
